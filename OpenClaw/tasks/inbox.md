# 任务收件箱

> 用途：收纳"将来要做"但还没开始的任务、想法、用户提过但没立即推进的请求。
> 位置：`~/.openclaw/tasks/inbox.md`（全局唯一，跨工作目录共享）。
> 写入：任何时候。
> 清理：开工时由 captain 评估"今天做什么"再决定升级或归档。
>
> OpenClaw 是常驻数字员工，任务可能涉及多个仓库 / 多种工作目录，请在每条任务里写清楚 **workdir 字段**，开工时按工作目录串行处理。

---

## 收件箱条目格式

```
- [ ] [YYYY-MM-DD] 简短描述 — 来源：用户对话 / Heartbeat / 周复盘 / lessons — 优先级：P0/P1/P2/P3
  - workdir：涉及的具体工作目录（或写 "global" 表示纯 Agent 任务）
  - 上下文：……
  - 验收：达成什么算完成（即使现在只能粗略写）
  - 阻塞：依赖什么前置（人 / 信息 / 工具 / 时间）
```

## 操作规则

- inbox 不是 backlog 黑洞，超过 3 个月没动的条目要在周复盘时显式决定：升级、降级、丢弃。
- 同一类需求出现 2 次以上 → 抽出成"重复请求"，下次直接升级。
- 优先级会变。变了写 `[P1→P0 原因：……]`，不要静默改字段。
- inbox 条目不允许并发执行。要做的事一旦开始，搬到 `tasks/in-flight.json`，inbox 这条标记 `[in-flight]`。
- 任务做完，inbox 这条标记 `[done @ YYYY-MM-DD → daily-log]`，正式记录放 daily-log。

---

## 不该写进 inbox 的事

- 一闪而过的奇思（不可执行的，写进笔记 app，不写这里）
- 用户随口一提但本人都没确认（先回去问一下再决定要不要落库）
- 用户已经明确说"不做"的事（直接拒绝，不留尾巴）
- 长期项目 / 整片愿景（这些放项目自己的 vision 文档，不进 inbox）
