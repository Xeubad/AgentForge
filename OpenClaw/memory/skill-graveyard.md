# Skill 坟场

> 用途：记录"试过但翻车"的 Skill、Plugin、Cron 配置。
> 写入触发：Skill 失败、产生意外副作用、被 lighthouse 拒绝、被用户拉黑。
> 价值：下次再有人推荐它、ClawHub 评分上来了、用户想"再试一次"，先查这里。

---

## 条目格式

```
### [YYYY-MM-DD] Skill 名 — 一句话结论

- 来源：clawhub / git / 内部 / 第三方
- 版本：……
- 触发场景：什么任务想用它
- 失败现象：表面发生了什么
- 根因：为什么失败（如果查清楚的话）
- 风险等级：低 / 中 / 高 / 极高
- 处置：拉黑 / 暂缓 / 等修复 / 替代为 ……
- 复活条件：什么情况下可以再考虑（或写"不会再考虑"）
```

---

## 示例（可保留作模板）

### [2026-05-19] auto_email_v2 — 拉黑，绝不再用

- 来源：clawhub（未签名）
- 版本：v2.0.3
- 触发场景：想自动发周报
- 失败现象：声称只发草稿，但实际触发了一次真实发送
- 根因：Skill 内部硬编码了一次 "first-run test send"，文档没说
- 风险等级：极高（不可逆 + 隐瞒副作用）
- 处置：拉黑；写进 `~/.openclaw/skills/allowlist.yaml` 的 denylist
- 复活条件：不会再考虑（即使官方修复，信任已破坏）

### [2026-05-19] doc_translator_v1 — 暂缓

- 来源：clawhub（已签名）
- 版本：v1.4.0
- 触发场景：把 README 翻成英文
- 失败现象：翻译质量勉强，但偶尔吞掉代码块
- 根因：未查证，疑似处理 markdown 时正则没考虑嵌套
- 风险等级：中
- 处置：暂缓使用
- 复活条件：等到 v2 或社区出现明确修复 changelog

---

## 维护原则

- 不删历史。一条 Skill 即使后来"复活"，旧条目也保留，加一条新条目标记复活原因。
- 失败原因如果没查清，**也要写**，标注"根因未明"——这本身就是有用信号。
- 同名不同版本的 Skill 算不同条目。
- 用户主动拉黑的 Skill 也归到这里，写明用户原话作为依据。
