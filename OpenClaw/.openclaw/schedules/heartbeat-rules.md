# Heartbeat 节制规则

Heartbeat 是 OpenClaw 的"自驱动"心脏：每 30 秒检查一次有没有活干。它是 OpenClaw 比其他 AI Agent 更强的地方，也是它最容易失控、最容易打扰用户、最容易在用户不在场时做错事的地方。

这份文件定义 Heartbeat 周期内**可以做什么、绝不能做什么、什么时候要主动汇报**。

---

## 1. Heartbeat 的合法用途

Heartbeat 只用来做四件事，超出这四件事的动作必须等用户对话指令：

1. **状态自检** — 读取 `tasks/in-flight.json`、检查未完成任务、识别可恢复点
2. **整理日志** — 把零散的执行记录合并进 `tasks/daily-log.md`
3. **检查到期** — 看 Cron 配置里有没有该触发的草稿（注意：只生成草稿，不发送）
4. **推送提醒** — 把"有什么待用户决策"的事项浮出到用户面前

允许的动作集（白名单）：

```yaml
allowed_actions:
  - read_file
  - grep_in_repo
  - tidy_logs               # 合并日志
  - generate_draft          # 只生成草稿，不发送
  - surface_pending         # 浮出待办给用户
  - check_skill_health      # 检查 Skill 是否还能正常加载
```

---

## 2. Heartbeat 绝对禁止动作（黑名单）

无论上下文如何，Heartbeat 不允许：

- 发送任何邮件 / IM / 通知（推送 OS notification 也算，除非用户显式开启）
- 提交、推送、发布、合并、回复 issue
- 调用任何付费 / 计费 API
- 安装 Skill / 插件 / 依赖
- 改写代码、改写配置（即使是 `.openclaw/` 自身的配置也不行）
- 修改 Cron / Heartbeat 自身参数
- 触发任何在 `irreversible-actions.md` 第 1 节里的动作
- 跨项目读取（不允许 Heartbeat 串走 `~/.openclaw/` 之外的别人项目）

---

## 3. 节流与去重

Heartbeat 每 30 秒一次，但**不等于每 30 秒都要做点什么**。默认应当大多数周期内"什么都不做"。

节流规则：

- 同一类提醒，**每小时**最多浮出 1 次
- 主动消息推送到用户，**每小时**全局上限 3 条（由 `config.yaml: max_proactive_messages_per_hour` 控制）
- 同一份 daily-log 整理，每 30 分钟最多一次
- 同一个失败的 Skill，**5 分钟**内不重复重试
- 连续 3 个 Heartbeat 周期产出相同结论 → 抑制，不再重复输出

---

## 4. 失败处理

Heartbeat 周期里跑的任何动作失败，处理流程：

1. 在 `tasks/daily-log.md` 写一条 `[heartbeat-failure]` 记录
2. 不重试同一个动作超过 1 次（除非有时间窗策略）
3. 如果同一类失败 24 小时内 ≥ 3 次：
   - 自动暂停该类自检
   - 切 `lighthouse` 写一份 incident 草稿
   - 主动浮出给用户
4. **从不**因为失败就降低安全规则，例如"为了让任务能跑就给 Skill 更多权限"——这是反模式

---

## 5. 主动汇报触发条件

只在以下情况，Heartbeat 才允许主动给用户推消息：

- **P0 风险**：发现可能的安全事件、敏感数据外泄迹象、第三方 Skill 异常
- **阻塞解除**：之前被等待用户确认的任务，等到了某个外部前置条件就绪
- **承诺时点到了**：用户说过"明天 9 点之前提醒我"
- **重大失败**：连续失败次数触发上限

不允许的主动汇报：

- "我做完了 XX"（这是任务收尾，由用户回到对话时再汇报）
- "我有个建议……"（除非用户主动问）
- "请问要不要 ……"（除非这是被卡住的任务的必要决策）
- "我闲着，给您整理了 ……"

主动汇报模板：

```
[Heartbeat]
触发原因：……（P0 / 阻塞解除 / 承诺到时 / 失败上限）
事实：……
建议下一步：……
该消息生成时间：……
```

---

## 6. 与 Cron 的边界

Heartbeat 是"短周期、轻量"机制；Cron 是"长周期、按计划"。两者职责不重叠：

| 维度 | Heartbeat | Cron |
|---|---|---|
| 周期 | 30 秒 | 分钟 / 小时 / 天 |
| 触发 | 自动 | 配置文件 |
| 默认权限 | 只读 + 整理 + 草稿 | 同 Heartbeat |
| 是否需要用户开启 | 默认开 | **默认关**，每条 Cron 都要用户显式确认 |
| 能否注册自己 | 不能 | 不能（注册 Cron 需要用户在对话里点头） |
| 失败后是否自动暂停 | 同类 24h 内 3 次后暂停 | 连续 3 次失败后暂停并通知 |

---

## 7. 用户关闭机制

用户必须随时能够：

- 一句话关停 Heartbeat：例如"关闭 Heartbeat" / "暂停自动检查"
- 一句话暂停某条 Cron：例如"暂停每日周报"
- 列出当前所有自动任务："列出所有定时任务"
- 删除某条调度："删除每日周报任务"

任何一条都必须立即生效，不需要重启、不需要进二级菜单。

---

## 8. 反模式（绝对不要做）

- "用户还没回我，我先猜一下他的意思继续做"
- "失败了，那我换个 Skill 再试"——尤其是换到没审核过的 Skill
- "我做了点事，反正改回去也方便"
- "用户说过类似的话，我把它当成永久授权"
- "趁用户不在，把仓库整理一下"
- "我把 Heartbeat 周期改成 5 秒，能更及时"——禁止自改周期
- "这条建议挺好的，我先做了，做完再告诉他"
