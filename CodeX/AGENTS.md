# Codex 高效开发规则

本文件是 Codex 项目级入口规则。Codex 会在开工前读取 `AGENTS.md`，并按从全局到项目、从上层目录到当前目录的顺序合并规则；更靠近当前目录的规则优先覆盖上层规则。

## 项目目标

- 本目录用于沉淀 Codex 的高效率工作配置。
- 目标不是堆积 prompt，而是让 Codex 在真实项目里更快进入状态、更少走弯路、更稳定执行。
- 默认输出语言为中文。
- 仓库是唯一事实来源；不在仓库内的背景、决策和约束，对 Codex 等于不存在。

## Codex 配置分层

- 全局个人规则：`~/.codex/AGENTS.md`。
- 临时全局覆盖：`~/.codex/AGENTS.override.md`，只用于短期覆盖，完成后删除。
- 项目规则：仓库内 `AGENTS.md`。
- 项目配置：可信项目内 `.codex/config.toml`。
- 自定义子代理：`.codex/agents/*.toml` 或 `~/.codex/agents/*.toml`。
- 可复用技能：`.codex/skills/<skill>/SKILL.md` 或 `~/.codex/skills/<skill>/SKILL.md`。
- 运行态状态：`tasks/`，包括进度、功能清单、交接摘要和质量快照。

## 核心原则

- 安全性 > 正确性 > 执行效率 > 表达风格。
- 优先做能闭环的问题解决，而不是做看起来聪明的输出。
- 优先给出可验证结果，而不是长篇解释。
- 不确定时明确说不确定，不猜。
- 人格和自定义 agent 只影响语气、协作方式和分工，不覆盖安全、正确性和验证要求。

## 开工流程

1. 读取当前 `AGENTS.md`、相关子目录 `AGENTS.md` 或 `AGENTS.override.md`。
2. 如果存在，读取 `tasks/progress.md`、`tasks/feature-list.json`、`tasks/session-handoff.md`。
3. 确认当前最高优先级未完成功能、已有 `in_progress` 项和 blocker。
4. 形成短期冲刺合同：本轮做什么、不做什么、验收标准、验证方式、失败时如何判断。
5. 同一时间只推进一个明确功能；已有 `in_progress` 时优先继续或显式调整状态。
6. 修改前先读相关文件，确认现状、边界和已有约定。

## 执行纪律

- 最小改动闭环：先做最小但完整的改动，让问题先被解决。
- 不顺手改无关代码，不把一次任务扩展成全面翻修。
- 发现现实与计划明显偏离时，立即停下并重新规划。
- 发现已有组件、工具函数、类型定义或配置时，优先复用。
- 涉及外部库、框架能力、OpenAI/Codex 配置或第三方 API 时，不猜；优先查官方文档。
- 禁止在没有依据时臆造业务规则、接口字段、错误处理策略或配置键。

## 验证与完成定义

- 没有验证，不算完成。
- 能运行测试就运行测试；不能运行时，明确说明未验证项、原因和风险。
- 标准启动路径或标准验证路径本身失败时，先修基础状态，不在坏基础上继续叠功能。
- `passing` 只能表示真实验证通过，必须有运行过的命令、结果或可复查证据。
- 不允许把“看起来应该可以”“没有明显报错”“我认为已完成”当成完成证据。
- 完成判定按三层检查：实现是否符合目标、验证是否真实运行、系统级行为是否满足用户路径。
- 核心功能未验证通过前，不做顺手重构、性能优化或风格清理。

## 子代理协作

Codex 通过 `.codex/agents/*.toml` 定义子代理，需要**显式调用**（不自动路由）。按任务类型选择合适的 agent：

| Agent | 定位 | 何时调用 |
|---|---|---|
| **rin** | 默认实现 + 审查 | 日常开发、代码审查、风险检查、需要直接结论时 |
| **haru** | 教学讲解 | 需要理解原理、架构取舍、分步解释时 |
| **en** | 严格审查 | 代码 review、架构审查、上线前检查、高风险变更 |
| **yuzu** | 排障推进 | 任务卡住、需要小步排错、环境问题、调试 |
| **aura** | 重构打磨 | 代码清理、命名一致性、UI 规范、可维护性提升 |
| **kurohomura** | 遗产清理 | 旧代码重构、技术债清理、需要趣味框架激励时 |

**协作模式**：

- **单 agent**：任务明确、范围小 → 直接调用最匹配的 agent
- **接力模式**：任务涉及多个阶段 → 按阶段切换 agent（如 `yuzu` 定位问题 → `rin` 修复 → `en` 审查）
- **并行模式**：大范围改动 → 多个 agent 各负责一个模块，最后由 `en` 汇总审查
- **审查模式**：实现完成后 → `en` 做最终审查，不通过则打回对应 agent 重做

**调用示例**：
- `用 rin 实现用户登录接口`
- `切到 en 审查一下这个 PR`
- `yuzu 帮我排查为什么测试挂了`
- `aura 把这个组件的命名和结构整理一下`

## 运行态状态

- 非琐碎任务开始前，更新 `tasks/feature-list.json` 或 `tasks/todo.md`。
- 多功能或长会话任务优先使用 `tasks/feature-list.json`，任何时候只能有一个 `in_progress`。
- 长会话、跨模块任务或上下文即将膨胀时，更新 `tasks/session-handoff.md`。
- 重要改动后可更新 `tasks/quality.md`，记录项目质量是否变强或变弱。
- 临时任务过程不写进 memory；长期可复用经验才写入 memory。

## 可观测性

- 过程可观测性：计划、范围、验收标准、排除项和评估理由要能被下一轮复查。
- 运行时可观测性：重要功能应能通过日志、错误上下文、健康检查、测试输出或命令结果说明系统实际做了什么。
- 失败信息应尽量包含修复方向，避免只有“失败了”而没有下一步线索。

## 干净收尾

- 收尾前确认标准验证是否运行；没运行要说明原因。
- 更新任务状态，不能留下未记录的半成品。
- 清理临时调试日志、临时文件、注释掉的废代码和无主 TODO。
- 记录下一步最佳动作，让下一轮不需要重新猜上下文。
- 如果仓库不是 Git 仓库或无法提交，明确说明不能提交，不假装完成提交。
- 当前目录不是 Git 仓库时，不强制提交；如果是 Git 仓库，每个功能或修改独立提交。

## 输出规则

- 默认先给结论，再补必要理由。
- 不堆无用背景，不重复显而易见的内容。
- 把重点放在：问题是什么、为什么这样改、验证了什么、还剩什么风险。
- 用户要求“少废话”“直接一点”时，收敛到高信噪比输出。

## 提交规范

- `feat:` 新功能。
- `fix:` 修复 bug。
- `refactor:` 代码重构。
- `style:` 样式调整。
- `perf:` 性能优化。
- `chore:` 构建、工具或配置更新。
