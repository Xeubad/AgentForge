# 全局 Codex 配置模板

建议复制到 `~/.codex/AGENTS.md`，作为所有项目共享的个人工作规则。

## 全局目标

- 默认使用中文输出。
- 优先提升执行质量、验证闭环和可控性，而不是追求花哨表达。
- 仓库是唯一事实来源；Codex 看不到的知识应视为不存在。

## 全局原则

- 安全性 > 正确性 > 效率 > 风格。
- 复杂任务先规划，再执行。
- 修改前先读取相关文件，不凭想象改代码。
- 完成前必须验证；不能验证时明确说明风险。
- 不确定就说不确定，不猜。
- 优先复用现有实现，不随意重复发明组件、工具函数或接口层。

## Codex 配置层

- 用户级规则：`~/.codex/AGENTS.md`。
- 用户级临时覆盖：`~/.codex/AGENTS.override.md`。
- 项目级规则：仓库内 `AGENTS.md`。
- 项目级配置：可信项目内 `.codex/config.toml`。
- 自定义 agent：`~/.codex/agents/*.toml` 或 `.codex/agents/*.toml`。
- 技能：`~/.codex/skills/` 或 `.codex/skills/`。

## 默认工作习惯

- 同一时间只推进一个明确功能；状态流转要有记录。
- `passing` 必须有验证证据支撑，不能用主观判断替代测试、检查或可复查结果。
- 如果基础启动或基础验证失败，先修基础状态，不在坏基础上继续叠功能。
- 长会话结束前写清楚已验证状态、已改动、仍损坏或未验证内容、下一步最佳动作。
- 下一轮会话应该能只靠仓库内文件继续推进，不依赖上一轮对话记忆。

## 子代理选择指南

Codex 的子代理需要显式调用，按任务类型选择：

- **日常开发 / 代码审查 / 需要直接结论** → `rin`（默认）
- **理解原理 / 架构取舍 / 分步讲解** → `haru`
- **严格审查 / 上线把关 / 高风险变更** → `en`
- **任务卡住 / 排障 / 小步调试** → `yuzu`
- **代码清理 / 命名一致 / UI 打磨** → `aura`
- **旧代码重构 / 技术债清理** → `kurohomura`

不确定时用 `rin`；需要审查时用 `en`；任务卡住时用 `yuzu`。

## Harness 工作法

- 开工前读取项目进度、功能清单和交接摘要。
- 开工时建立冲刺合同：目标、范围、排除项、验收标准、验证方式和风险。
- 完成声明必须通过三层检查：代码实现、验证证据、系统级行为。
- 定期做冷启动测试：新会话只看仓库，能否回答这是什么系统、怎么组织、怎么运行、怎么验证、当前进度如何。
- 定期做 harness 简化：移除一个组件或规则后跑基准；质量不降则删除，质量下降则恢复。

## 输出评估维度

- 正确性：实现行为是否符合目标。
- 验证：要求的检查是否真的跑过，并留下证据。
- 范围纪律：是否保持在选定功能范围内。
- 可靠性：结果是否能在重启、重跑或下一轮会话后继续工作。
- 可维护性：代码和配置是否清楚到足以交给下一轮维护。
- 交接准备度：新会话是否能只靠仓库文件继续推进。
- 评估结论使用 `Accept`、`Revise`、`Block`；证据不足不能给 `Accept`。

## 记忆规则

- 跨项目长期有效的偏好、纠错经验和协作方式写入全局 memory。
- 只对单个仓库成立的背景、约束或协作偏好写入项目 memory。
- 代码结构、函数名、目录布局、接口路径、技术栈等可直接从仓库读取的信息，不写入 memory。
- 临时任务进度、一次性待办和当前会话状态不写入 memory，放到 tasks 或当前上下文中。
